# Search top service

Сервис для отображения топа поисковых запросов за последние 5 минут.
Создан в рамках тестового задания для команды поиска/ранжирования Wildberries

## Быстрый старт

### Через Docker Compose

```bash
docker compose up -d
```

Сервис поднимается на `http://localhost:8080`

Накидать мок-данные:
```bash
go run ./cmd/producer/...
```

## Makefile

`make up` -поднять все сервисы через docker compose

`make down` - остановить и удалить контейнеры

`make run` - запустить сервис локально без Docker 

`make build` - билд проекта

`make test` - запустить юнит-тесты 

`make produce` - накидать мок-данные в NATS 

## API

### GET /api/v1/top

Возвращает топ-N поисковых запросов за последние 5 минут

**Параметры:**
- `n` — количество элементов (по умолчанию 10, максимум 100)

**Запрос:**
```bash
curl "http://localhost:8084/api/v1/top?n=5"
```

**Ответ:**
```json
{
  "items": [
    {"query": "кроссовки асикс", "count": 142},
    {"query": "электрочайник",  "count": 98},
    {"query": "айфон 15",       "count": 76}
  ],
  "total": 3
}
```

---

### POST /api/v1/stoplist

Добавить слово в стоп-лист. Слово немедленно исчезает из топа без перезапуска сервиса

```bash
curl.exe -X POST http://localhost:8084/api/v1/stoplist \
  -H "Content-Type: application/json" \
  -d '{\"word\":\"viagra\"}'
```

---

### DELETE /api/v1/stoplist/{word}

Убрать слово из стоп-листа

```bash
curl -X DELETE http://localhost:8084/api/v1/stoplist/viagra
```

### GET /metrics

Prometheus-метрики

curl http://localhost:8084/metrics

## Контракт данных (NATS)

**Subject:** `search.events`

**Stream:** `SEARCH`

**Пример сообщения:**
```json
{
  "query":      "кроссовки асикс",
  "user_id":    "u-abc123",
  "session_id": "s-xyz789",
  "timestamp":  1737123456
}
```

**Обоснование полей:**

`query` string - основная единица подсчёта

`timestamp` int64 (Unix UTC) - определяем, попадает ли событие в 5-минутное окно; int64 для того, чтобы не парсить строки

`user_id` string - антинакруточное поле

`session_id` string - дополнительный сигнал накрутки

## Архитектура

### Sliding window

Временное окно 5 минут разбито на **300 бакетов по 1 секунде**. Каждую секунду фоновая горутина:

1. Сдвигает указатель на следующий бакет и очищает его (самый старый автоматически дропается)
2. Собирает все 300 бакетов в map
3. Применяет стоп-лист
4. Пересчитывает топ-100 и кэширует результат

**Почему не один большой map с TTL на каждый ключ**

Удаление по TTL требует прохода по всей map каждую секунду — O(n) по числу уникальных запросов;

Подход с бакетами будет O(bucket_size), и он не зависит от числа уникальных запросов

### Топ-N: min-heap

Для получения топ-N из M уникальных запросов используется min-heap размера N;

Сложность: O(M log N) вместо O(M log M) при полной сортировке

### Стоп-лист: sync.Map

O(1) на чтение и запись, нет лока для читателей. Обновляется на лету

### Кеш топа

Топ пересчитывается **раз в секунду** фоновой горутиной и кэшируется;

Каждый HTTP-запрос берёт кэш через `sync.RWMutex`, заново ничего не считаем

## Компромиссы и решения

### Проблемы в постановке задачи

**1. "Последние 5 минут"**

Не совсем понятная формулировка "последние 5 минут" - начиная с чего, есть ли округление

Решение: скользящее окно ровно 300 секунд от текущего момента

**2. Формат payload не задан**

Контракт определил самостоятельно (см.выше), ключевое решение — `timestamp` как int64 вместо строки

**3. "Аномальные всплески"**

Неясно, какое кол-во запросов считать аномалией;

Решение: обнаружение накрутки по `user_id` + `session_id` на уровне консьюмера.

Порог настраивается через конфиг без перекомпиляции

**4. Данные теряются при рестарте**

Сознательный компромисс в пользу производительности и простоты.

Альтернатива по типу снапшотов на диск или Redis плохо влияют на латентность;

Для виджета "сейчас ищут" потеря данных при рестарте некритична, тк окно заполнится за 5 минут

## Мониторинг

`http://localhost:8080/metrics`

Ключевые метрики:


`search_events_processed_total` - сколько событий обработано из NATS

`search_events_dropped_total{reason}` - дропнутые события по причине

`search_top_requests_total` - нагрузка на ручку топа

`search_http_duration_seconds` histogram - латентность HTTP запросов

`search_active_queries_total` gauge - уникальных запросов в текущем окне
